# Documentation Style Standard

## Requirement

All documentation in this repository must use ASD-STE100 Simplified Technical English. This standard applies to each Markdown document and each prose document in the repository.

## Sentences and paragraphs

- Use a maximum of 20 words in a procedure sentence.
- Use a maximum of 25 words in a description sentence.
- Use a maximum of 6 sentences in a paragraph.
- Use a vertical list for a series or a set of steps. Put one item on each line.

## Grammar and voice

- Write in the active voice.
- Write in the present tense. Use the past tense or the future tense only when necessary.
- Write an instruction in the imperative mood. For example, write "Click Save."
- Write one instruction in one sentence.
- Do not omit an article. Always write a, an, or the.
- Avoid an -ing verb form when a simple form is possible. A Technical Name that contains -ing stays as it is.

## Words and terms

- Use one term for one concept in the whole document. For example, pick "NexusApp" or "the app" and keep it. Do not change the name.
- Use a maximum of 3 words in a noun cluster. Unpack or hyphenate a longer cluster.
- Use "must" for a requirement. Use "can" for a capability.
- Do not use these words: shall, should, may, might, could.

## Simple words

- Use simple words. Use words like these: use, start, stop, show, get, make, do, before, to, with.
- Do not use these words: utilize, leverage, initiate, terminate, obtain, prior to, in order to, via, ensure, e.g., i.e., etc.
- Replace a complex word with a simple word:
  - Write "before" in place of "prior to".
  - Write "to" in place of "in order to".
  - Write "with" in place of "via".
  - Write "make sure" in place of "ensure".
  - Write "for example" in place of "e.g.".
  - Write "that is" in place of "i.e.".
  - Remove "etc." List the items instead.

## Tone

- Do not use an idiom, marketing language, humor, a rhetorical question, or an exclamation mark.
- Do not use an em-dash or an en-dash. Use a period, a comma, parentheses, or a hyphen.

## Warnings and cautions

- Put a warning or a caution before the step that it applies to.
- Put the warning or the caution on its own labeled line.

## Exempt categories

These categories do not follow this standard. Keep the text exactly as written:

- legal text
- license text
- trademark notices
- direct quotes
- Technical Names
- code
- commands
- file paths

## Technical Names

A Technical Name stays exactly as written. Do not change a Technical Name to follow a rule. These names are Technical Names:

- Star Citizen
- NexusApp
- RS Signal Decoder
- Data.p4k
- Game.log
- global.ini
- WebView2
- GitHub
- Inno Setup
- Windows SmartScreen
- Easy Anti-Cheat

A file path, a command, a key name, and a product name are also Technical Names.

## Example

Before (not Simplified Technical English):

> In order to leverage the auto-scan capability, the RS value should be obtained via the decoder, and the game files (e.g. Data.p4k) may be utilized to ensure accuracy.

After (Simplified Technical English):

> To use the auto-scan feature, get the RS value with the decoder. NexusApp can read the game files, for example Data.p4k, to make the result accurate.

## Exempt text example

A trademark notice is exempt. Keep it exactly as written:

> Star Citizen®, Roberts Space Industries® and Cloud Imperium® are registered trademarks of Cloud Imperium Rights LLC.
